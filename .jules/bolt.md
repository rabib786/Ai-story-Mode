## 2024-05-17 - Array Allocation Optimizations
**Learning:** Found instances where `.slice().reverse().findIndex(...)` and `.slice().reverse().find(...)` were used on large data structures (`chatHistory`). This pattern creates a full shallow copy and then a full reversed copy of the array on every operation just to find the last occurrence of an item, resulting in unnecessary O(N) memory allocation and processing time.
**Action:** When searching backward in arrays (especially long lists like chat histories), implement and use O(1) space, O(N) time custom `findLast` or `findLastIndex` utilities (via reverse loops) to avoid garbage collection overhead and reduce latency. Remember that `findLastIndex` returns the actual forward index, removing the need for `length - 1 - reversedIndex` math.
## 2025-03-08 - React.memo Defeated by Inline Closures
**Learning:** Found instances where `React.memo` components (`UserMessageBubble`, `ModelMessageBubble`) were passed inline arrow functions in the `chatHistory.map` loop. This caused unnecessary re-renders on every parent render, completely defeating the memoization.
**Action:** When mapping over large lists to render memoized components, ensure all callback props are memoized (using `useCallback`) and passed by reference. Push closures down into the child component where possible (e.g., passing `message.id` down instead of capturing it in an inline function).
## 2025-03-08 - Deep Clone Performance Cost
**Learning:** Found an instance where `JSON.parse(JSON.stringify(prev))` was used to create a deep clone of a large data structure (`chatHistory`) just to modify a single element. This serialization/deserialization approach is a massive performance bottleneck.
**Action:** When updating a specific element in an array within React state, prefer a shallow clone of the array (`[...prev]`) combined with a shallow copy of only the specific object being modified (`{ ...prev[index] }`). This avoids the high CPU and memory overhead of full deep cloning, improving operation speed significantly (e.g., ~80x faster in a 1000-message benchmark).
## 2025-03-08 - React.memo Defeated by Inline Closures in Lists
**Learning:** Found instances where `React.memo` was missing on list item components (`ScenarioCard`, `CharacterCard`), and their parents were passing inline arrow functions (e.g., `onSelect={() => onSelectScenario(scenario)}`) inside `.map()` loops. This creates new function references on every parent render, causing unnecessary re-renders of all list items even if their data hasn't changed.
**Action:** When rendering lists of items, wrap the item component in `React.memo()`. Pass stable callback references down (e.g., `onSelect={handleSelect}`) and let the child component invoke the callback with its specific data (e.g., `onClick={() => onSelect(item)}`). This ensures `React.memo` works correctly and prevents O(N) re-renders when parent state updates.
## 2025-03-08 - Avoid `.map().lastIndexOf()` for Finding Elements
**Learning:** Found instances where `.map()` was combined with `.lastIndexOf()` to find the index of an element matching multiple conditions in an array (`chatHistory`). This approach creates an entire intermediate array, leading to unnecessary memory allocations and redundant iterations.
**Action:** When finding the last matching item in an array in environments where `findLastIndex` is not natively supported (like ES2022 TypeScript without generic helpers), use a custom reverse `for` loop. A reverse `for` loop is >900x faster for large arrays because it short-circuits. Wrap this within a `useMemo` block in React to avoid re-calculating on every render.
